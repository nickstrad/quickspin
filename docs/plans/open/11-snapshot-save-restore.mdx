# 11 — Save, restore, and warm starts

Depends on: 07 (06's reconciler assumed).

## Summary

Add the "save sandboxes" half of your platform description: stop a sandbox while
preserving its filesystem as a named snapshot, create new sandboxes *from* a snapshot,
and measure what this does to startup latency. Mechanically: `docker commit`-based
snapshots owned by tenants, new lifecycle verbs (`save`, restore-by-create), and a
first honest latency benchmark (cold create vs snapshot create).

<KeyPoint label="Name the promise precisely">
  This plan preserves the <em>filesystem</em>, not memory or running processes. The
  API says `save`, not `pause`, and the docs must say so too — blurring that line is
  how platforms end up with users who think their running Python process survives.
</KeyPoint>

## Industry context

Persistence and fast resume are where sandbox platforms differentiate hardest. E2B
pauses sandboxes and resumes them with **memory state intact** (Firecracker VM
snapshots); Daytona persists workspaces; Modal builds images from snapshotted layers.
Your Docker-commit version preserves **filesystem only, not memory or running
processes** — a real and useful capability (an agent's installed deps and written code
survive), but a different promise than E2B's, and the plan requires naming that
difference in your API docs rather than blurring it. Filesystem-vs-memory snapshotting
is precisely the boundary where microVMs earn their keep, which sets up plan 14.

## What you'll learn

- Docker image mechanics for real: layers, commit, what balloons image size, why
  writable-layer growth matters.
- Lifecycle design when states multiply: what is `saved`? (This plan: a snapshot is an
  artifact, and the sandbox that produced it is destroyed on save — no paused state.)
- Benchmarking honestly: percentiles over means, measuring from API call to guest
  `/healthz`, separating pull/create/start/ready phases.
- Cleanliness of restored state: what leaks through a snapshot (files in `/tmp`,
  credentials the agent wrote) — a small threat-model addendum.

### Concept: `docker commit` freezes the writable layer into a new image

A running container is an image's read-only layers plus one writable layer where every
file the agent created, modified, or deleted lives (deletion is a whiteout entry, not
reclaimed space). `docker commit` seals that writable layer as a new layer on top of
the stack and names the result an image — which is why a snapshot restores as an
ordinary create: `Spec.Snapshot` just resolves to an image reference instead of a
registry tag. Two consequences matter for your implementation. Size is cumulative:
every commit adds a layer, and a snapshot of a snapshot carries both writable layers,
so the byte quota is measured against real `docker image inspect` sizes, not
guesses. And the commit captures *configuration* as well as files — entrypoint,
env, and any bind-mount expectations from the source container — which is why the plan
strips those at commit time (`docker commit` accepts config overrides) so the image
stays a portable filesystem, restorable by any backend.

### Concept: save is a distributed write, and the order is the correctness

"Save" performs three effects on two systems: commit the image (Docker), record the
snapshot row (store), destroy the source sandbox (Docker). The plane can crash between
any two, so the order is chosen to make every partial state recoverable. Record the
snapshot *before* destroying the source: a crash after commit-and-record but before
destroy leaves a sandbox that should be gone — which the reconciler already knows how
to converge (plan 06's desired-vs-observed loop, now with `saving` as a desired state
that resolves to destroyed). The reverse order would leave a committed image with no
row — an orphan the reconciler cannot distinguish from garbage. That same
reconciler-driven GC is the delete path: `DELETE /v1/snapshots/{id}` removes the row,
and the reconciler notices an image with no referencing row and removes it, so image
deletion is retried until it succeeds rather than trusted to one HTTP handler.

### Concept: a benchmark you can defend

"Snapshots make startup faster" is only meaningful as a measured claim with a stated
method. The honest shape: measure from the API call to the guest reporting healthy
(the latency an agent actually feels), record per-phase timestamps (pull, create,
start, guest-ready) so you can say *where* time went, run enough iterations that a
percentile means something, and report p50/p95 rather than a mean that one slow pull
can poison. The comparison must also be fair: cold create *plus the setup step the
snapshot amortizes* (the `pip install`) versus snapshot create — comparing bare cold
create to snapshot create measures nothing, since the snapshot's whole value is the
work it pre-paid. The CSV is checked in because a number you can't reproduce next
month is a number you don't have.

<Callout tone="warning" title="Snapshots copy secrets as faithfully as code">
  Everything the agent wrote is in the image: tokens saved to `~/.netrc`, keys in
  `/tmp`, shell history. A snapshot shared across trust boundaries would leak them.
  Tenant scoping contains this for now, but the threat-model addendum in this plan
  exists to record the exposure — write it, don't skip it.
</Callout>

## Design and interfaces

```text
POST /v1/sandboxes/{id}/save   body: {"name": "deps-installed"}
     → 202; sandbox transitions stopping→saved-and-destroyed; snapshot recorded
GET  /v1/snapshots             list (tenant-scoped)
DELETE /v1/snapshots/{id}
POST /v1/sandboxes             spec gains {"snapshot": "deps-installed"} as an
                               alternative to {"image": ...} (exactly one required)
```

Schema: `snapshots(id, tenant_id, name, image_ref, size_bytes, created_at,
source_sandbox_id)`. Runtime interface gains `Snapshot(ctx, id) (imageRef, error)`;
`Spec` gains `Snapshot string`.

Committed decisions: save destroys the source sandbox (a snapshot is a checkpoint you
resume *from*, not a paused thing you resume — E2B-style pause is out of scope and the
API name says `save`, not `pause`); snapshots count against a per-tenant total-bytes
quota; guest binary is re-injected on restore so snapshots never bake in a stale guest.

## Hint steps

<HintSteps>
  <Hint
    title="Prove commit round-trips by hand before writing Go"
    check="A manually committed container restores with docker run and the file you wrote inside it is present."
  >
    In a shell: create a sandbox, `docker exec` a file into it, run
    `docker commit CONTAINER scratch/test`, then `docker run` the new image and
    check the file.
    Inspect the committed image's config (`docker image inspect`) — note that the
    source's entrypoint and env came along. That is what your `Snapshot`
    implementation must strip; the SDK's `ContainerCommit` takes a
    `container.Config` override for exactly this.
  </Hint>
  <Hint
    title="Implement Snapshot behind the Runtime interface"
    check="A Go integration test can call Snapshot on a running sandbox and create a new sandbox whose Spec.Snapshot names the returned imageRef, and the file written before the save is visible in the new sandbox."
  >
    `Snapshot(ctx, id)` resolves the quickspin ID to a container via the label filter
    (plan 02's helper), calls `ContainerCommit` with a repo/tag you control (e.g.
    `quickspin-snapshot:<id>` plus a `quickspin.managed` image label for GC), and
    returns the image ref. On the create path, `Spec.Snapshot` short-circuits image
    resolution — and skips the registry pull, since the image is local by
    construction. Validate exactly-one-of image/snapshot at the API boundary, not in
    the runtime.
  </Hint>
  <Hint
    title="Order the save handler: commit, record, then destroy"
    check="Reading the handler top to bottom shows the snapshot row inserted before any destroy call, and the 202 response carries the snapshot's identity."
  >
    The handler: resolve tenant + sandbox, admission-check the byte quota (size comes
    from `ImageInspect` after commit — reject and clean up the image if over), insert
    the row, then mark the sandbox's desired state destroyed and let the reconciler
    do the destroy. Doing the destroy inline is tempting but wrong twice: it makes
    the 202 slow, and it duplicates convergence logic the reconciler owns.
  </Hint>
  <Hint
    title="GC is a set difference the reconciler already knows how to take"
    check="Deleting a snapshot row by hand (sqlite3, DELETE FROM snapshots...) leads the next reconcile pass to remove the image, visible in docker images."
  >
    Each pass: list images carrying your snapshot label (`ImageList` with a filter),
    list `image_ref` values from the store, remove images in the first set but not
    the second (`ImageRemove`). Same shape for over-quota enforcement if you choose
    to GC rather than only reject at admission. Watch the plan 06 trap: the diff must
    be store-driven, so a Docker-side listing failure must abort the pass, never be
    treated as "no images exist".
  </Hint>
  <Hint
    title="Re-inject the guest on restore rather than trusting the snapshot"
    check="TestRestoredSandboxHasFreshGuest passes: after bumping the guest version and restoring an old snapshot, the guest health endpoint reports the new version."
  >
    Whatever mechanism plan 07 used to place the guest binary into a fresh sandbox
    must run on the snapshot-create path too — restore is just create, so if guest
    injection lives in the create flow (not baked into base images) you may already
    be done; the test is what proves it. The failure mode being prevented: a
    six-week-old snapshot resurrecting a six-week-old guest API.
  </Hint>
  <Hint
    title="Instrument phases in the benchmark, then aggregate percentiles"
    check="hack/bench-11.sh produces a CSV with one row per iteration and columns per phase, and snapshot-create rows lack the setup-phase cost that cold rows carry."
  >
    Timestamp with `date +%s%N` (or do the whole loop in Go — plan 03 already fought
    shell timing precision) around each phase: create request returns, state reaches
    running, guest `/healthz` first succeeds, and for the cold arm, the `pip install`
    exec completes. 20 iterations each arm; sort each column and take index 10 and 19
    for p50/p95 (`sort -n | awk` is enough). Alternate arms rather than running 20
    then 20, so daemon warmup doesn't bias one side.
  </Hint>
  <Hint
    title="Diagnosing the likely failure: the crash test loses the snapshot or leaks the sandbox"
    check="TestSaveIsRecordedBeforeDestroy passes: killing the plane between commit and destroy, then restarting it, leaves exactly one snapshot row, a restorable image, and no source sandbox."
  >
    Reuse the plan 06 crash-point technique (an env-var hook such as
    `QUICKSPIN_CRASH_AFTER=snapshot-record` that exits the process) rather than
    timing kills. If after restart the snapshot is missing, your handler recorded it
    after initiating destroy — reorder. If the source sandbox survives forever, the
    desired-state write never happened or the reconciler doesn't treat that state —
    check which by reading the row: desired state tells you whether the handler or
    the reconciler is the broken half. If the image exists but restore 404s, the row
    insert and the commit disagree on the ref — log both at commit time and compare.
  </Hint>
</HintSteps>

## Definition of done

Integration tests (`make test-docker`):

- `TestSaveThenRestorePreservesFiles` — write file, install a package, save, create
  from snapshot, both survive; source sandbox is gone.
- `TestRestoredSandboxHasFreshGuest` — guest healthy and version-current after restore.
- `TestSaveIsRecordedBeforeDestroy` — kill the plane between commit and destroy;
  reconciler converges without losing the snapshot (crash point in `validate-06`
  style).
- `TestSnapshotTenantScoping` — tenant B cannot restore from A's snapshot (404).
- `TestSnapshotByteQuota` — over-quota save rejected with a typed code.
- `TestSnapshotGC` — deleting the DB row leads the reconciler to remove the image.

Benchmark: `hack/bench-11.sh` runs green and its CSV shows snapshot-create p50
meaningfully below cold-create-with-setup p50 (the number itself is the artifact).

Deliberately untested: memory/process state (explicitly not preserved), concurrent
saves of one sandbox, snapshot export off-host.

## Solo-developer tradeoffs

E2B's pause/resume snapshots RAM in hundreds of milliseconds via Firecracker; that is
a company-scale feature sitting on a custom hypervisor stack. `docker commit` gives you
80% of the agent-workflow value (durable environments, warm dependency caches) with one
API call, and — more importantly for your goals — forces you to articulate the
filesystem/memory distinction that makes the microVM pitch legible in interviews. The
byte quota exists because snapshots are the first thing in this system that silently
consumes disk forever; commercial platforms meter and bill it, you cap it.
