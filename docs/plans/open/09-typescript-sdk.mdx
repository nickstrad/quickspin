# 09 — TypeScript SDK

Depends on: 07, 08.

## Summary

Create `sdk/typescript/` (`@quickspin/sdk`): a typed client for the control-plane API
with the E2B-style ergonomic surface agents expect — `Sandbox.create()`, `sbx.exec()`,
`sbx.files.*`, `sbx.kill()`. Per your direction, **Claude generates this SDK; your job
is review and contract-testing, not hand-writing TypeScript**. Your Go time in this
plan goes into the other deliverable: `GET /v1/openapi.json` served by the control
plane, making the API formally described.

<KeyPoint label="Division of labor">
  The agent writes the client; you write the spec and the tests that hold the client
  accountable. Reviewing generated code against a contract you authored is a different
  skill from writing the code — and it is the one this plan teaches.
</KeyPoint>

## Industry context

The SDK is the actual product surface of platforms like E2B — nobody curls these APIs;
agents import `@e2b/code-interpreter` and call three methods. Conventions this SDK
copies deliberately: constructor/env-var API key resolution (`QUICKSPIN_API_KEY`),
resource-handle objects rather than bare functions, typed error subclasses
(`QuotaExceededError`, `SandboxNotFoundError`) so harnesses can branch on failure type,
and async iteration for streamed exec output. Serving an OpenAPI document is how
commercial platforms keep N SDKs honest against one API; yours will do the same work
in plan 10.

<Compare>
  <CompareItem eyebrow="What you build" title="Spec-anchored, contract-tested">
    The OpenAPI document is authoritative; a Go test pins it to the real routes, and a
    Vitest suite pins the SDK to a live server. Drift in any of the three surfaces
    fails a build.
  </CompareItem>
  <CompareItem eyebrow="The anti-pattern" title="SDK as folklore">
    A hand-maintained client with no spec and mocked tests can diverge from the API
    for months without anyone noticing — the mocks keep passing against an API shape
    that no longer exists.
  </CompareItem>

## What you'll learn

(Go-side, where your focus is:) API self-description — writing an OpenAPI 3.1 document
for your own endpoints and discovering where your API is irregular; error-envelope
stability as a compatibility contract. (Review-side:) judging an SDK you didn't write
against a spec you did.

### Concept: the spec is a contract only if something enforces it

An OpenAPI document that nobody checks is documentation, and documentation rots. This
plan wires two enforcement points around the spec. On the Go side, a test enumerates
the mux's registered routes and diffs them against the paths in `openapi.yaml` — add an
endpoint without documenting it (or vice versa) and `make test` fails. On the SDK side,
the contract suite runs every generated method against a **real** control plane, so the
spec's request and response shapes are exercised end to end, not assumed. Writing the
spec is also where you discover your API's irregularities: the endpoint that returns a
bare array where every other returns an envelope, the one field that is camelCase amid
snake_case. Finding those now is the point — plan 10 ports the same contract to Python,
and every irregularity you keep costs twice.

### Concept: ndjson streaming through an async iterator

The exec-stream endpoint sends newline-delimited JSON over a chunked HTTP response.
The SDK's job is to turn that byte stream into `for await (const ev of ...)` — which
means owning three details that libraries usually hide. First, chunk boundaries are
arbitrary: one network read can contain half an event or three of them, so the parser
buffers bytes, splits on `\n`, and keeps the trailing partial line for the next read.
Second, liveness is the whole point: events must be yielded as they arrive, not
accumulated until the response ends — an SDK that buffers the body defeats streaming
while still passing every non-timing test. Third, teardown: if the consumer breaks out
of the loop early, the underlying reader must be cancelled so the HTTP connection and
the server-side exec are released. The Definition of done's liveness assertion exists
to catch the second failure, which is by far the most common.

### Concept: typed errors as the machine-readable half of the API

Plan 05 committed to an error envelope with a stable `error.code`. The SDK converts
those codes into subclasses of a common `QuickspinError` so a harness can write
`catch (e) { if (e instanceof QuotaExceededError) ... }` instead of matching message
strings. Two rules make this durable. The mapping is 1:1 and closed over the API's
published codes — an unknown code maps to the base class rather than crashing, so a
new server code degrades gracefully in an old SDK. And messages are for humans only:
the moment a caller parses `e.message`, the message text becomes API surface and can
never be reworded. This is the TypeScript twin of the Go sentinel-error convention
from plan 02 — same contract idea, different language mechanism.

<Callout tone="note" title="instanceof and Error subclasses">
  When reviewing the generated errors, check that each subclass either targets ES2015+
  or calls `Object.setPrototypeOf(this, new.target.prototype)` in its constructor —
  the classic transpilation bug makes `instanceof QuotaExceededError` silently false,
  and every typed-error contract test would catch it only if the tests assert with
  `instanceof`, not `e.code`.
</Callout>

## Design and interfaces

SDK surface (generated code must match this):

```ts
const sbx = await Sandbox.create({ image: "python:3.12-slim", ttlSeconds: 600 });
const res = await sbx.exec("python", ["-c", "print(1)"], { timeoutMs: 30_000 });
res.exitCode; res.stdout; res.stderr;               // buffered mode
for await (const ev of sbx.execStream("make", ["test"])) { ... } // streamed mode
await sbx.files.write("/app/main.py", code);
const text = await sbx.files.read("/app/main.py");
await sbx.keepalive();
await sbx.kill();                                    // idempotent, like the API
Sandbox.connect(id)                                  // reattach to an existing sandbox
```

Committed decisions: zero runtime dependencies (native `fetch`), works on Node 20+,
ndjson stream parsing in-house; errors map 1:1 from the API's `error.code` values.

## Hint steps

<HintSteps>
  <Hint
    title="Start with the spec, and let it audit your API"
    check="openapi.yaml validates (npx @redocly/cli lint openapi.yaml or the Swagger editor) and lists every /v1 route with its error envelope schema."
  >
    Before any TypeScript exists, write `openapi.yaml` by walking the plan 05/07
    handlers. Define the error envelope once under `components.schemas` and reference
    it from every non-2xx response — if you find yourself unable to, that endpoint's
    errors are irregular and should be fixed in Go now. Keep a running list of every
    irregularity you hit; each is a candidate API fix before generation starts.
  </Hint>
  <Hint
    title="Pin the spec to the mux with a Go test"
    check="Adding a fake route to the mux (or deleting a path from the YAML) makes the new test fail in make test with a readable diff of the two path sets."
  >
    Serving the JSON is the easy half: embed the YAML with `go:embed`, convert once at
    startup (`sigs.k8s.io/yaml` or `gopkg.in/yaml.v3` into a generic map, then
    `encoding/json`), and hand it to a handler. The sync test is the valuable half:
    build the set of documented `method path` pairs by decoding the YAML's `paths`
    map, build the set of registered patterns from wherever your routes are declared
    (this is the nudge to keep a single route table both the mux and the test read),
    and assert set equality.
  </Hint>
  <Hint
    title="Review the generated client top-down, spec in hand"
    check="You can state, for each of the eight surface methods, which OpenAPI operation it calls and where its types come from — and package.json plus a grep for import statements confirm zero runtime dependencies."
  >
    Review order that works: `package.json` (no deps, Node 20 engines), the error
    module (one subclass per documented `error.code`, `instanceof` intact — see the
    callout), the request core (base URL + key resolution from `QUICKSPIN_API_KEY`,
    one place that throws on non-2xx), then the resource methods. Compare each request
    and response type against the spec, not against the server code — the spec is the
    contract you agreed to enforce.
  </Hint>
  <Hint
    title="Scrutinize the ndjson parser hardest"
    check="A scratch test that feeds the parser one event split across two chunks, and two events in one chunk, yields exactly the right events in both cases."
  >
    The shape to expect: an async generator wrapping `res.body.getReader()`, a
    `TextDecoder` with `stream: true`, a string buffer that splits on newline and
    retains the final partial segment, and a `finally` that calls `reader.cancel()`.
    Anything that awaits `res.text()` or joins chunks before parsing has silently
    destroyed streaming — it will pass every test except the liveness one.
  </Hint>
  <Hint
    title="Make the test harness boot a real plane"
    check="make test-sdk-ts from a clean checkout starts the plane, runs vitest, exits non-zero on failure, and leaves no server process behind."
  >
    Shell script shape: `mktemp -d` for the SQLite file, start the control plane in
    the background, poll its health endpoint until ready (curl in a loop with a
    timeout), mint a key with the plan 08 admin CLI, export `QUICKSPIN_API_KEY` and
    the base URL, run `npx vitest run`, and kill the server in a `trap ... EXIT` so
    cleanup survives test failure. Vitest picks up the env vars without help.
  </Hint>
  <Hint
    title="Assert liveness with two clocks, and typed errors with instanceof"
    check="The execStream test records a first-event timestamp measurably earlier than command completion, and each typed-error test passes via expect(err).toBeInstanceOf(...)."
  >
    For liveness, stream a command that prints then sleeps (`sh -c 'echo hi; sleep
    2'`) and assert the first event arrives well before total elapsed time — a
    buffering SDK collapses the two. For errors, drive each failure for real: a
    garbage key for `AuthenticationError`, a tenant quota of zero (plan 08 admin CLI)
    for `QuotaExceededError`, exec after `kill()` for `SandboxNotFoundError`. Use
    Vitest's `rejects.toBeInstanceOf` so message wording stays out of the contract.
  </Hint>
  <Hint
    title="Diagnosing the likely failure: the liveness test times out or flakes"
    check="With the fix in place, the first stream event consistently arrives within a few hundred milliseconds while the command still runs, on repeated make test-sdk-ts runs."
  >
    Three suspects, in order. The SDK buffers (see the parser hint) — verify with a
    raw `curl -N` against the endpoint: if curl shows lines arriving live, the server
    is fine and the SDK is the problem. The server doesn't flush — plan 07's handler
    must call `http.Flusher.Flush()` after each event; without it Go buffers small
    writes. Or the test itself is wrong — asserting on wall-clock gaps that are too
    tight for CI. Fix by asserting ordering (first event before process exit) rather
    than absolute milliseconds.
  </Hint>
</HintSteps>

## Definition of done

`make test-sdk-ts` passes, including at minimum:

- `create → exec → files round-trip → kill` happy path.
- `execStream` yields output events before command completion (liveness assertion).
- Typed errors: bad key ⇒ `AuthenticationError`; over-quota ⇒ `QuotaExceededError`;
  killed sandbox exec ⇒ `SandboxNotFoundError`.
- `kill()` twice resolves both times.
- `Sandbox.connect()` reattaches and can exec.

Go-side: the route/OpenAPI sync test is in `make test`.

Deliberately untested: browser runtime, proxy environments, retry/backoff policy
(explicitly none in v0 — retries interact with idempotency keys and deserve their own
decision later).

## Solo-developer tradeoffs

Commercial SDKs are fully codegen'd (Stainless/Fern) with retries, pagination, and
telemetry baked in. A hand-shaped (if machine-written) client with zero dependencies is
easier to read end-to-end — which matters when the SDK is also a learning artifact and
a resume exhibit. The real discipline kept from industry: contract tests against a live
server, so the SDK can never silently drift from the plane.
