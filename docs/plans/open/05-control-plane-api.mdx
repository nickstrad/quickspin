# 05 â€” Control plane: HTTP API and durable state

Depends on: 04.

## Summary

Wrap the runtime in a long-running Go HTTP service â€” the control plane. It owns sandbox
records in SQLite, exposes a versioned JSON API (`/v1/sandboxes`...), assigns lifecycle
states, and supports idempotent creation via client-supplied idempotency keys. The CLI
becomes a client of the API instead of calling the runtime directly. From this plan on,
Quickspin is a service, not a tool.

## Industry context

This is the piece users literally point their SDKs at: E2B's and Daytona's public REST
APIs look strikingly like the surface below (create with a spec, poll state, exec,
files, delete). Two industry patterns are load-bearing here. First, **desired state
lives in a database, actual state lives in the runtime**, and they are allowed to
disagree â€” reconciling them is plan 06's job; this plan makes the disagreement
representable. Second, **idempotency keys** (as popularized by Stripe's API) are how
every serious platform makes "the client retried a create after a network blip" not
produce two billed sandboxes.

## What you'll learn

- Production-shaped Go HTTP services: `net/http` with Go 1.22+ method routing,
  middleware, graceful shutdown, structured logging (`log/slog`), request IDs.
- SQLite from Go: schema migrations, transactions, and why "state machine in a
  database" is a transactional problem.
- The error/logging conventions at a second layer: `ControlPlaneError` with `Wrap`
  above the runtime's errors, one-log-per-error (the handler that converts an error to
  an HTTP response is the propagation stop, so it logs), child loggers per component.
- Designing a lifecycle state machine and rejecting illegal transitions.
- API error design: stable machine-readable error codes vs prose.

### Concept: the database records intent, the runtime records reality

Until now, "does the sandbox exist" had one answer: ask Docker. This plan introduces a
second answer â€” the SQLite row â€” and the design hinges on which one is written first.
The create handler inserts a `pending` record *before* touching the runtime, then
transitions it to `running` or `failed` based on what actually happened. That ordering
is what makes crashes representable instead of catastrophic: if the process dies
between the insert and the Docker call, restart finds a `pending` row with no
container â€” a fact a future reconciler (plan 06) can see and act on. Written the other
way around (Docker first, row second), a crash produces a container no database knows
about: an invisible, unbilled, undeletable leak. The database row is not a cache of
Docker's state; it is the authoritative statement of what *should* exist, and the two
are allowed to disagree until something reconciles them.

### Concept: a state machine is only as strong as its narrowest gate

The lifecycle states are cheap to define; the discipline is that every state change in
the codebase flows through one gate. Split the problem into two layers, per the
code-structure reference. The *legality* question â€” may a sandbox go from `stopped` to
`running`? â€” is a pure function, `canTransition(from, to State) bool`, exhaustively
table-tested with no database in sight. The *atomicity* question is transactional: the
`transition(id, from, to)` helper must read the current state, check legality, and
write the new state inside one transaction, so two concurrent requests cannot both
observe `running` and both "win" a transition out of it (the classic lost-update
race). Scattered `UPDATE sandboxes SET state=...` statements bypass both layers at
once, and each one is a future bug in which a `failed` sandbox quietly resurrects.

### Concept: idempotency keys turn retries into reads

A create request can succeed on the server while the response is lost to a network
blip. The client cannot tell "failed" from "succeeded, reply lost", so a correct
client retries â€” and without protection, the retry creates a second sandbox. The
idempotency key inverts the responsibility: the client names the *operation* (a random
key in the `Idempotency-Key` header), and the server promises that one key maps to at
most one sandbox. Mechanically that is the `idempotency_keys(key, sandbox_id, ...)`
table: on create, look the key up; a hit returns the existing sandbox's current record
(making the retry a read), a miss creates and records the mapping. The subtle cases
worth deciding consciously: a request with no key is a plain create every time, and
the lookup-or-insert should share the create's transaction â€” though truly concurrent
retries racing one key are explicitly out of scope until plan 08's quota work makes
transactions load-bearing anyway.

<StudyPrompt question="Why does the key map to a sandbox ID rather than a stored copy of the original response?">
  Ask what the retry should see if the sandbox failed after creation. Replaying a
  cached 201 body would report a healthy sandbox that no longer exists. Mapping to the
  ID and re-reading the record means the retry gets the truth as of now â€” the key
  deduplicates the side effect, not the response bytes. (Stripe caches responses, but
  its resources are immutable charges, not long-lived machines.)
</StudyPrompt>

### Concept: the handler is where errors stop being Go values

Below this plan, failures travel as wrapped sentinels (`errors.Is(err,
runtime.ErrNotFound)`). Above it, they travel as JSON with a machine-readable `code` â€”
because SDK code will branch on `"sandbox_not_found"`, and prose messages must be free
to improve without breaking anyone. The handler is the propagation stop: it is the one
place that inspects the chain with `errors.Is`, chooses a status (404, 422, 409),
writes the envelope through `WriteError`, and logs the full diagnostic once. Nothing
below it logs-and-returns (double logging), and nothing above it exists. Keeping the
`ErrorResponse` DTO and the `WriteJSON`/`WriteError` helpers in `internal/httpapi` â€”
owned by neither server nor client â€” is what lets plans 09, 10, and 17 consume the
same wire format instead of re-deriving it.

## Design and interfaces

Lifecycle states and legal transitions:

```text
pending -> running -> stopping -> stopped
pending -> failed
running -> failed
```

HTTP surface (JSON bodies; errors as `{"error": {"code": "...", "message": "..."}}`):

```text
POST   /v1/sandboxes              create (honors Idempotency-Key header)
GET    /v1/sandboxes              list
GET    /v1/sandboxes/{id}         inspect
DELETE /v1/sandboxes/{id}         destroy (idempotent: 204 even if already gone)
POST   /v1/sandboxes/{id}/exec    run a command (buffered result)
PUT    /v1/sandboxes/{id}/files?path=/abs/path    write (body = content)
GET    /v1/sandboxes/{id}/files?path=/abs/path    read
GET    /v1/sandboxes/{id}/dir?path=/abs/path      list
```

SQLite schema (minimum): `sandboxes(id, state, spec_json, runtime_ref, created_at,
updated_at)` and `idempotency_keys(key, sandbox_id, created_at)`.

Committed decisions:

- Create is **synchronous** in this plan (request returns when the sandbox is running
  or failed). Plan 06 revisits this honestly; note it now as a known simplification.
- The store is behind a small Go interface so tests can use `:memory:` SQLite. Per the
  concurrency reference's store rules: typed reads, records by value in and out,
  `ErrNotFound` + zero value for missing keys, and **no locking or transaction policy
  of its own** â€” invariants that span calls (state transition + event append, quota
  check + insert in plan 08) are the caller's transaction.
- Now that plan 15 commits prod to Postgres, two of its costs move here where they are
  nearly free: the schema ships as **versioned migration files** (`migrations/NNNN_*.sql`,
  embedded, applied at startup) rather than create-if-missing, and it avoids
  SQLite-isms Postgres will reject (app-generated string IDs, no serial PKs, ISO-8601
  UTC text timestamps mapping cleanly to `timestamptz`). The store test suite is
  written as `storetest.Run(t, factory)` from day one so plan 15 adds a second caller
  instead of an extraction refactor.
- State transitions happen in one place (a `transition(id, from, to)` helper inside a
  transaction) â€” never as scattered UPDATEs. The legality check itself is a pure
  function, `canTransition(from, to State) bool`, with a table test enumerating the
  full matrix (code-structure reference, candidate #3).
- `internal/httpapi` owns the response dance (`WriteJSON`, `WriteError`) and the
  `ErrorResponse` DTO; handlers never hand-roll status-plus-JSON, `WriteError` sets the
  body's code from the status it writes, and 204 writes no body. This package is shared
  wire format â€” the SDKs (09/10) and the worker client (17) will consume the same DTO,
  which is why it lives in neither server nor client.

<Callout tone="system" title="The one ordering that is not negotiable">
  Never let an HTTP handler touch Docker without recording intent in the database
  first. Every other decision in this plan can be revisited; this ordering is the
  foundation plan 06's reconciler stands on.
</Callout>

## Definition of done

Store unit tests (`make test`), authored inside `storetest.Run`:

- `TestIllegalTransitionRejected` â€” e.g. stoppedâ†’running fails atomically.
- `TestIdempotencyKeyReturnsSameSandbox`.
- `TestCanTransitionMatrix` â€” pure table over every (from, to) pair.

- `TestCreateInspectDestroyFlow` â€” JSON contract, status codes, error envelope shape.
- `TestDeleteIsIdempotent` â€” second DELETE also 204.
- `TestRuntimeFailureMarksSandboxFailed` â€” fake runtime error â‡’ record in `failed`,
  429/5xx taxonomy correct.

End-to-end validation script `hack/validate-05.sh` against the real service + Docker:
starts `quickspin serve`, drives createâ†’execâ†’file write/readâ†’destroy with `curl`,
asserts on JSON with `jq`, kills the server mid-run and confirms a restart still lists
the surviving sandbox record. Exits 0 only if all checks pass and no labeled containers
leak.

Deliberately untested: concurrent creates racing one idempotency key, auth (plan 08),
TLS, request body size limits.

## Solo-developer tradeoffs

Commercial planes are Postgres-backed, horizontally scaled, and async-first (create
returns `pending` immediately; you poll or subscribe). SQLite plus synchronous create
keeps this a single deployable binary with zero infra â€” the right call for one
developer â€” and the store interface plus the state machine mean the Postgres/async
migration later is mechanical, not architectural. The one habit not to defer: never let
an HTTP handler touch Docker without recording intent in the database first. That
ordering is what plan 06 builds on.
