# 17 — Control plane / worker split and host heartbeats

Depends on: 15, 16 (07's guest architecture assumed).

## Summary

Break the monolith: sandbox execution moves into `quickspin-workerd`, a daemon that
runs on each host, exposes the `Runtime` operations over an authenticated internal
HTTP API, registers itself with the control plane, and heartbeats. The plane —
which can now run anywhere, since it no longer needs `/dev/kvm` — keeps a `hosts`
table, places new sandboxes on a healthy host with capacity, and (via the reconciler)
declares hosts dead when heartbeats stop, failing their sandboxes. This is the plan
that makes "monitor live cloud hosts" a property of the system rather than of your
SSH habits.

## Industry context

This is the canonical control-plane/data-plane split: Kubernetes' apiserver/kubelet,
Nomad's server/client, and (per their engineering posts) how E2B and Fly place microVMs
on host fleets. The mechanisms you're building are the textbook trio — registration,
heartbeats with a dead-host threshold, and capacity-aware placement — plus the rule
that makes distributed state survivable: the database's desired state is authoritative,
workers hold no durable truth, and a dead worker's sandboxes are *failed, not
migrated* (sandboxes are stateful; honest platforms kill them and let clients recreate
from snapshots — exactly why plan 11 exists).

## What you'll learn

- Designing an internal service-to-service API and its auth (static bearer token per
  host from the plane's admin CLI; mTLS noted as the industry answer and deferred).
- Failure detection theory made concrete: heartbeat interval vs death threshold,
  why "no heartbeat" means *unknown* not *dead*, and what a flapping host does to
  naive logic.
- Placement as a pure function: `place(spec, []Host) (hostID, error)` — trivially
  testable, no I/O (the code-structure reference doc's pattern applied for real). But
  the pure function only *chooses*; the *reservation* is a check-then-act hazard — two
  concurrent creates can both pick a host's last slot. Committing the placement re-checks
  capacity inside the insert's transaction (`FOR UPDATE` on the host row), the same
  fold-the-check-into-the-reserve rule as plan 08's quota, one layer down.
- Both process boundaries follow the error reference: workerd serves the shared
  `httpapi.ErrorResponse`; the plane's worker client logs the DTO and mints a
  `ControlPlaneError` (never a fake wrap), and `internal/worker` gets its own
  `WorkerError` type with the standard 30 lines.
- The reconciler generalized across two layers: converging sandboxes *and* hosts.

### Concept: a missed heartbeat is absence of evidence, not evidence of death

A heartbeat is the only signal the plane has about a worker, and it is a weak one: when
heartbeats stop, the plane cannot distinguish a crashed workerd from a network
partition, a GC pause, or a plane-side ingestion hiccup. The host's sandboxes may be
running perfectly. That ambiguity is why the design uses *two* thresholds rather than
one. At 30 seconds (three missed 10-second beats) the host becomes `unready`: the
plane stops sending new work — a cheap, reversible act, because refusing placement on
a healthy host costs a little capacity while placing onto a dead one loses a
customer's sandbox. Only at 2 minutes does the plane commit to `dead` and take the
expensive, effectively irreversible action of failing the host's sandboxes and
emitting events. The gap between the thresholds is the flapping budget: a host whose
network blips for 40 seconds goes unready, resumes heartbeating, and returns to
`ready` with nothing killed. Naive single-threshold logic turns every blip into a
mass sandbox failure — `TestFlappingHostRecovers` exists to pin the correct behavior.

### Concept: choosing a host is pure; claiming a slot is not

`place(spec, []Host)` is deliberately a function with no I/O: filter to hosts that are
`ready`, match the requested backend and arch, have headroom, then pick the
least-loaded. Every placement policy question becomes a table test. But the function
runs on a snapshot of the hosts table, and between "I read that host A has one slot
left" and "I inserted a sandbox on host A", another request can do the same read.
Both pass the pure check; the host ends up over capacity. This is the classic
check-then-act race, and the fix is the same one plan 08 used for quotas: fold the
check into the reservation by re-verifying capacity *inside* the transaction that
inserts the sandbox row, with the host row locked (`SELECT ... FOR UPDATE`). The pure
function remains the policy; the transaction is the arbiter.
`TestConcurrentPlacementNeverOverfillsHost` must fail if the transactional re-check
is deleted — if it still passes without it, the test is not exercising the race.

<KeyPoint label="Invariant">
  The pure placement function may be wrong (stale snapshot); the reservation
  transaction may not be. Capacity is enforced where the row is written, never where
  it is read.
</KeyPoint>

### Concept: workers hold no durable truth

The split only works because of a state-ownership rule: the database's desired state
is authoritative, and workerd is a stateless executor over its local runtime. Workerd
persists nothing it cannot rebuild — on boot it re-registers, reports what is actually
running (discovered from labels, plan 02's rediscovery mechanism paying off across a
process boundary), and takes orders. That is why `kill -9` on workerd is a legitimate
test rather than a catastrophe, and why a dead host's sandboxes are *failed, not
migrated*: the sandbox's memory and disk lived only on that host, so pretending to
move it would be lying to the client. Failing fast with an event, and letting the
client recreate from a plan 11 snapshot, is what honest platforms do. The same rule
explains `no_capacity` as a fast typed error rather than a queue: queueing is durable
plane-side state with its own fairness and expiry semantics — real future work, not a
default to back into.

<StudyPrompt question="Why does the plane never SSH into a worker?">
  SSH is a human's escape hatch, not a program's API: unversioned, untyped,
  screen-scraped, and offering root. The moment the plane depends on SSH, worker
  behavior can no longer be faked in a unit test, auth can't be scoped to specific
  verbs, and every operation has a hidden dependency on shell details of the host
  image. Two typed HTTP surfaces (plane API and workerd API) keep every cross-process
  interaction testable with a fake client and revocable with a token — which is
  exactly what `TestDeadHostFailsItsSandboxes` and friends rely on.
</StudyPrompt>

## Design and interfaces

```text
quickspin-workerd (per host):
  POST /register     at boot: host name, arch, backend(s), capacity (max sandboxes)
  Serves the Runtime surface over HTTP: create/inspect/list/destroy per plan 02/14
  (workload ops still go plane -> guest directly, unchanged from plan 07)
  POST /heartbeat    every 10s to the plane: running count, disk/mem headroom

Control plane:
  hosts(id, addr, arch, backend, capacity, state{ready,unready,draining,dead},
        last_heartbeat_at, registered_at)
  Placement: filter ready+backend+capacity, pick least-loaded; no host ⇒ create
  fails fast with a typed "no_capacity" error (backpressure, not queueing — queueing
  is future work and saying so is the design decision).
  Reconciler additions: no heartbeat > 30s ⇒ unready (no new placements);
  > 2m ⇒ dead ⇒ its sandboxes -> failed, events emitted.
  Admin: quickspin admin host list|drain|token ; GET /v1/status gains host summary.
  sandboxes gains host_id.
```

Committed decisions: the plane never SSHes to workers — everything flows through the
two HTTP surfaces; worker tokens are minted per host and revocable; `drain` stops
placement but leaves running sandboxes to finish or expire (the graceful-maintenance
primitive plan 18 automates); local dev runs plane + one workerd in Lima so the split
is exercised daily, not only in prod.

<Callout tone="warning" title="Thresholds are compared against the DB clock you wrote, not wall time you remember">
  Every heartbeat handler writes `last_heartbeat_at`; every reconciler pass computes
  `now - last_heartbeat_at`. Both must use the same injected clock in tests, or the
  threshold tests become flaky sleeps. Design the state-transition logic as a pure
  function of `(host, now)` from the start — the fake-clock tests in the definition
  of done assume it.
</Callout>

## Hint steps

<HintSteps>
  <Hint
    title="Extract workerd as a shell, not a rewrite"
    check="A second binary builds from cmd/quickspin-workerd, and curl against its create endpoint with a hardcoded token starts a sandbox on the local runtime."
  >
    The first question is what workerd *doesn't* contain: no store, no reconciler, no
    business rules — just `net/http` handlers that decode a request DTO, call the
    existing `Runtime`, and encode `Info` or the shared `httpapi.ErrorResponse`. Add
    a `cmd/quickspin-workerd/main.go` and reuse the runtime packages untouched. Auth
    is one middleware doing a constant-time compare
    (`crypto/subtle.ConstantTimeCompare`) of the `Authorization: Bearer` value
    against the configured token. If you find yourself copying plane logic into
    workerd, stop — that logic belonged in a shared package or in the plane.
  </Hint>
  <Hint
    title="Registration and heartbeat are upserts keyed on host identity"
    check="Starting workerd twice produces one hosts row, and last_heartbeat_at visibly advances every ~10s while it runs."
  >
    Decide the host's stable identity (its name is fine at this scale) and make
    `/register` an upsert (`INSERT ... ON CONFLICT ... DO UPDATE`) so re-registration
    after a crash refreshes `addr`, capacity, and state rather than duplicating the
    row — a workerd restart must be routine. The heartbeat loop in workerd is a
    `time.Ticker` posting the running count and headroom; on the plane it is a
    single-row UPDATE of `last_heartbeat_at` plus the reported gauges. Registration
    should also flip an unready host straight back to ready.
  </Hint>
  <Hint
    title="Write place as a pure function and table-test it before any wiring"
    check="TestPlacementPrefersLeastLoaded, TestPlacementFiltersArchAndBackend, and TestPlacementNoCapacityIsTypedError pass with no database or HTTP in sight."
  >
    Signature per the design: `place(spec, []Host) (hostID, error)`. Filter (ready,
    backend, arch, headroom) then select least-loaded, with a deterministic tie-break
    (lowest ID) so tests don't flake. The empty-result case returns the typed
    `no_capacity` error, constructed here so the API layer only translates it.
    Getting these tables green first means the later wiring bugs can only be in
    wiring.
  </Hint>
  <Hint
    title="Make the reservation transactional, then try to break it"
    check="TestConcurrentPlacementNeverOverfillsHost admits exactly M of N creates — and starts failing when you comment out the FOR UPDATE re-check."
  >
    Inside the create transaction: `SELECT ... FROM hosts WHERE id = $1 FOR UPDATE`,
    recount that host's active sandboxes, and abort with `no_capacity` if the slot
    vanished between the pure choice and now. The concurrency test is N goroutines
    calling create against one host of capacity M through the real store (this is
    what plan 15's Postgres is for), asserting exactly M successes and N−M typed
    failures. Run it with `go test -race -count=10` — a race that shows up one run
    in five is still a race. Then do the deletion experiment in the check: a
    concurrency test that cannot fail proves nothing.
  </Hint>
  <Hint
    title="Drive host state transitions from an injected clock"
    check="TestMissedHeartbeatsMarkUnreadyThenDead passes by advancing a fake clock past 30s and 2m — no time.Sleep anywhere in the test."
  >
    Give the reconciler a `now func() time.Time` (or a small Clock interface) instead
    of calling `time.Now` inline, and factor the decision into a pure function of
    `(host, now)` returning the target state. Then the threshold tests are tables:
    beat at t=0, advance to t=31s, expect unready; advance to t=2m1s, expect dead
    plus the sandbox-failing side effects via a fake worker client.
    `TestFlappingHostRecovers` is the same machinery with a heartbeat landing at
    t=90s — assert ready again and, critically, that no destroy call ever reached
    the fake client.
  </Hint>
  <Hint
    title="Dead-host handling is a state change plus events, never a worker call"
    check="TestDeadHostFailsItsSandboxes marks only that host's sandboxes failed, emits an event per sandbox, and the fake worker client records zero calls to the dead host."
  >
    When a host crosses the dead threshold: update its sandboxes to `failed`, emit
    events, done. Do not try to destroy through the dead worker — it is unreachable
    by definition, and the runtime-level orphan sweep (plan 06, now running inside
    workerd) collects the corpses if the host ever returns. `drain` is the gentler
    sibling: a state that placement filters out while everything running continues —
    `TestDrainStopsPlacementKeepsRunning` pins both halves.
  </Hint>
  <Hint
    title="Cut over Lima and prod to two processes"
    check="hack/validate-17.sh boots plane + workerd in Lima, a sandbox lands via registration and placement, and on the Hetzner host systemctl shows both units green."
  >
    Local first: a make target or script that starts both binaries with a dev token
    so the split is exercised daily. For prod, plan 16's cloud-init gains a second
    systemd unit (`quickspin-workerd.service`, `After=quickspin.service` if it
    registers at boot) and the deploy script restarts both with the health gate
    still keyed on `/v1/status` — which now also asserts one ready host. Mint the
    worker token with `quickspin admin host token` and put it in the worker's env
    file, mode 600, same discipline as plan 16.
  </Hint>
  <Hint
    title="Diagnosing the likely failure: the kill -9 scenario misbehaves"
    check="In hack/validate-17.sh, after kill -9 the host reaches dead within the thresholds, its sandbox is failed with an event, creates fail fast with no_capacity — and a workerd restart re-registers and garbage-collects the orphan container."
  >
    Three distinct bugs hide in this one scenario. If the host never goes dead: the
    reconciler is comparing against a stale `now` or the interval between reconciler
    passes exceeds the threshold window — log heartbeat age each pass and watch it
    grow. If creates hang instead of failing fast: the plane's worker client lacks an
    `http.Client` timeout and is waiting on a socket that will never answer — every
    cross-process client needs a deadline. If the orphan container survives the
    workerd restart: the sweep is filtering by a label the plane-era create wrote but
    the workerd-era create doesn't (or vice versa) — `docker ps -a --filter
    label=quickspin.managed=true` inside Lima shows the truth. Use generous
    test-config thresholds (say 3s/10s via env) so the script runs in seconds, but
    keep the 10s/30s/2m production values as the documented defaults.
  </Hint>
</HintSteps>

## Definition of done

Unit tests (`make test`, fake clock, fake worker client):

- `TestPlacementPrefersLeastLoaded`, `TestPlacementFiltersArchAndBackend`,
  `TestPlacementNoCapacityIsTypedError` (pure-function table tests).
- `TestMissedHeartbeatsMarkUnreadyThenDead` — clock-driven, both thresholds.
- `TestDeadHostFailsItsSandboxes` — with events, and no effect on other hosts'
  sandboxes.
- `TestFlappingHostRecovers` — heartbeat resumes before the dead threshold ⇒ ready
  again, nothing was killed.
- `TestDrainStopsPlacementKeepsRunning`.
- `TestConcurrentPlacementNeverOverfillsHost` — N concurrent creates against one
  host with capacity M admit exactly M (the placement twin of plan 08's quota-race
  test; must fail if the transactional re-check is removed).

Integration `hack/validate-17.sh` (Lima, real plane + real workerd):

- Boot both; workerd registers; sandbox lands on it and works end-to-end.
- `kill -9` workerd: within the thresholds the host goes dead, its sandbox is
  `failed` with an event, and creates now fail fast with `no_capacity`.
- Restart workerd: re-registers, placement resumes; the killed sandbox's container
  is garbage-collected (the runtime-level orphan sweep from plan 06 still works
  through the worker).

Deliberately untested: two workerds racing one sandbox ID (single-owner-by-placement
is asserted by design, not by lease — recorded as the known gap vs the reference
doc's ownership-lease stage), mTLS, placement under thousands of hosts.

## Solo-developer tradeoffs

Nomad and Kubernetes solve this with raft-replicated servers, ownership leases, and
work queues; you run one plane instance (Neon is the only HA piece) and skip leases
because single-plane placement can't double-assign. That is a real correctness
dependency on "exactly one control plane" — written down, and the first thing to fix
if this ever became multi-plane. Failing dead hosts' sandboxes instead of migrating
them is not a shortcut, though: it is what the big platforms do, and snapshots are
the recovery story.
