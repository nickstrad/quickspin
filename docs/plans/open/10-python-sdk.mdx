# 10 — Python SDK

Depends on: 09.

## Summary

Create `sdk/python/` (`quickspin` package): the same client surface as the TypeScript
SDK, in idiomatic Python (sync-first, context managers, `httpx`). As with plan 09,
**Claude generates it; you review and gate on contract tests**. The learning payload
for you is smaller and different: watching whether your API survives a second language
untouched — every place the Python SDK needs a workaround is an API bug to fix in Go.

## Industry context

Python is the default language of agent frameworks (LangChain, CrewAI, OpenAI/Anthropic
tool loops), so a sandbox platform without a Python SDK doesn't exist to half the
market — E2B and Daytona both lead with Python. The industry pattern exercised here is
**API-first symmetry**: both SDKs derive from the same OpenAPI contract and pass
equivalent contract suites, which is the only scalable answer to N languages. Where
TypeScript is async-native, Python platforms typically ship sync-first with an async
variant later; this plan follows that.

## What you'll learn

- Reading your API through a second language's eyes: naming (`ttlSeconds` vs
  `ttl_seconds`), error-code stability, streaming ergonomics (`for ev in
  sbx.exec_stream(...)` over generators).
- Keeping two contract suites equivalent — and noticing when one suite asserts
  something the other forgot.
- Light Python packaging (`pyproject.toml`, `uv`) as consumer, not expert.

### Concept: the second language is a free API review

One consumer proves an API is usable; the second proves it is *regular*. Every place
the Python client needs special handling — a field that must be renamed at the
boundary, an endpoint whose error shape differs, a response the TypeScript SDK quietly
tolerated — is a defect that was always there and only became visible under a second
set of conventions. The wire format stays exactly what the OpenAPI document says
(`ttlSeconds` on the JSON, because plan 09 froze it); each SDK owns the translation to
its language's idiom (`ttl_seconds=` as a Python keyword argument, mapped in one place
at the request boundary). The discipline this plan asks for is to *log* every friction
point instead of silently absorbing it into the client — that list, sorted into
"Python fix" versus "Go API fix", is a deliverable, and the Go fixes are the ones that
pay off forever.

### Concept: context managers encode ownership, not just cleanup

`with Sandbox.create(...) as sbx:` is Python's way of making a lifetime lexical: the
`__exit__` method runs when the block ends, *including when the block raises*, which is
exactly the guarantee an agent harness needs so a crashed task doesn't strand a
sandbox. But the same syntax must not always mean "kill on exit" — `connect` attaches
to a sandbox someone else created, and attachment is not ownership. The clean design
gives the object an ownership flag set by its constructor path: `create` returns an
owning handle whose exit kills; `connect` returns a non-owning handle whose exit only
closes the HTTP client. Because `kill()` is idempotent (the API promised that in plan
05), the owning path can kill unconditionally in `__exit__` without special-casing
"already dead". This create-owns / connect-borrows split is the documented behavior
the contract tests pin.

### Concept: sync streaming is a generator over a live response

Where the TypeScript SDK exposed an async iterator, the Python SDK exposes a plain
generator: `for ev in sbx.exec_stream(...)`. `httpx` does the transport work — 
`client.stream("POST", url) as response` keeps the body open, and
`response.iter_lines()` handles the chunk-boundary buffering that plan 09 had to do by
hand. The subtlety that remains is lifetime: a generator body only runs while the
caller iterates, and the `with client.stream(...)` block inside it must stay open for
the whole loop — so the generator wraps the stream context and yields from inside it.
If the caller abandons the loop early, generator finalization closes the response and
frees the connection. Buffering bugs look identical to plan 09's: everything passes
except the liveness test, which is why the pytest suite repeats that assertion.

<StudyPrompt question="Why sync-first when the TypeScript SDK is async?">
  Ask who calls each SDK. TypeScript on Node is async by construction — blocking the
  event loop is not an option. Python agent harnesses are overwhelmingly plain
  scripts and sync frameworks; a sync client keeps the plan 12 demo readable, and an
  async variant can wrap the same wire logic later. Shipping async-first would force
  every consumer into an event loop to run one command — ergonomics backwards.
</StudyPrompt>

## Design and interfaces

```python
with Sandbox.create(image="python:3.12-slim", ttl_seconds=600) as sbx:
    res = sbx.exec("python", ["-c", "print(1)"], timeout_s=30)
    res.exit_code; res.stdout; res.stderr
    for ev in sbx.exec_stream("make", ["test"]):
        ...
    sbx.files.write("/app/main.py", code)
    text = sbx.files.read("/app/main.py")
# context-manager exit kills the sandbox

Sandbox.connect(sandbox_id)   # reattach without killing on exit unless asked
```

Committed decisions: `httpx` as the only runtime dependency; typed exceptions mirror
plan 09's names (`QuotaExceededError`, `SandboxNotFoundError`, `AuthenticationError`);
key from `QUICKSPIN_API_KEY`; the context manager kills on exit for `create` but not
for `connect` (attachment ≠ ownership — document this).

    check="Grepping the package for ttlSeconds finds it only in the request-serialization code, and grepping for ttl_seconds finds it only in the public surface and tests."
  >
    The wire names stay camelCase because the spec froze them; the Python surface is
    snake_case because PEP 8 says so. The translation belongs in one place — where
    the request dict is built — not sprinkled through the codebase. If the generated
    client instead changed the wire names, that is a contract violation the pytest
    suite would catch against the live plane; fix the SDK, never the spec.
  </Hint>
  <Hint
    title="Scrutinize __exit__ for the ownership split"
    check="A scratch test shows create's context manager calls kill on normal exit and on an exception raised inside the block, while connect's exits without any kill request."
  >
    Look for a single `Sandbox` class with an owning flag (or two constructor paths
    setting `_kill_on_exit`). `__exit__(self, exc_type, exc, tb)` must kill
    unconditionally when owning — not only when `exc_type is None` — and must return
    a falsy value so exceptions propagate; a `__exit__` returning `True` silently
    swallows the harness's errors, which is a bug worth hunting for explicitly.
  </Hint>
  <Hint
    title="Verify streaming is a generator over an open httpx stream"
    check="A quick REPL session shows the first event from exec_stream of a print-then-sleep command arriving while the command is still sleeping."
  >
    The shape: a generator function that opens `with self._client.stream("POST", ...)
    as response:` and yields `json.loads(line)` from `response.iter_lines()`, mapping
    non-2xx to the typed exceptions before yielding anything (use
    `response.read()` on the error path — a streaming response's body is not
    pre-loaded). A version that calls plain `.post()` and splits the finished body
    passes everything except liveness.
  </Hint>
  <Hint
    title="Port the suite test-by-test and write the parity table as you go"
    check="pytest -x against a booted plane passes, and every Vitest test name has a row in the parity table pointing at its pytest twin or an explained gap."

    Python-only additions (context-manager kill/no-kill, stream liveness) get their
    own named tests from the Definition of done; add each parity-table row as you
    port, not at the end.
  </Hint>
  <Hint
    title="Diagnosing the likely failure: the sandbox survives the with block"
    check="After the fix, test_context_manager_kills_on_exit passes including its raising variant, and listing the tenant's sandboxes after the block shows none."
  >
    If `test_context_manager_kills_on_exit` fails only in the raising variant, the
    generated `__exit__` guards the kill behind `exc_type is None` — ownership means
    cleanup on *every* exit. If it fails in both variants, check whether the test
    entered the manager at all: `Sandbox.create(...)` without `with` never calls
    `__exit__`, and a helper that returns an already-entered object breaks the
    protocol. Confirm from the plane's request log whether DELETE was ever issued —
    that tells you client bug versus test bug in one look.

## Definition of done

`make test-sdk-py` passes with the pytest equivalents of every plan 09 contract test,
plus:

- `test_context_manager_kills_on_exit` — sandbox is gone after the `with` block, even
  when the block raises.
- `test_connect_does_not_kill_on_exit`.
- `test_stream_yields_before_completion`.

Cross-SDK parity check: a short doc table (in the plan's closing notes) mapping each TS
contract test to its pytest twin, with gaps explained — parity is asserted by review,
not tooling, and saying so is part of done.

Deliberately untested: asyncio variant, Python versions below 3.11, Windows.

## Solo-developer tradeoffs

Commercial platforms generate both SDKs from the spec in CI and publish to npm/PyPI on
tags with semver guarantees. You keep the packages in-repo and unpublished until the
plan 12 demo needs to import them (a git or path dependency is fine for a learning
platform and avoids premature registry ceremony). Skipping asyncio-first is deliberate:
sync code keeps the demo harness readable, and the API's streaming already proves the
hard part works.
